# ADR: Next.js Dictionary Architecture

> **Status:** Proposed  
> **Date:** 2026-07-12  
> **Replaces:** Classic Nuxt offline CSV dictionary approach  
> **See also:** `docs/adr/lp-classic-dictionary-architecture.md` (Classic reference)

---

## Context

The Classic Nuxt app uses client-side CSV dictionaries loaded into IndexedDB. This has two fundamental problems for the Next.js web app:

1. **L1 coverage gap** — dictionaries only exist for English-based lookups. A Spanish-speaking user learning Japanese gets English definitions machine-translated on the fly, losing nuance and accuracy.
2. **Large downloads** — CSV files are 10-50MB each, unsuitable for web first-load.

Since the Next.js app is web-only (not offline) and the GO React Native app handles the offline use case, we have the freedom to use a server-side architecture.

---

## Decision

**Layered dictionary lookup via a Python backend endpoint, with LLM translation for non-English L1s and LLM generation as fallback for words not in any CSV dictionary.**

### Architecture

```
Browser (Next.js)
  │  tap word "食べる" (L1=es, L2=ja)
  ▼
POST /api/python/dictionary/lookup
  { text: "食べる", l2: "ja", l1: "es" }
  │
  ▼
Python Backend (new endpoint: app_dictionary.py)
  │
  ├─ 1. Load appropriate CSV dictionary for L2 (in-memory, cached)
  │     ├─ zh → cedict
  │     ├─ ja → edict
  │     ├─ ko → kengdic
  │     ├─ ru → open_russian
  │     └─ *  → wiktionary_csv
  │
  ├─ 2. Search: exact match → lemma match → fuzzy match
  │     Result includes `match_type: "exact" | "lemma" | "fuzzy"`
  │
  ├─ 3. If found AND L1 == "en":
  │     → Return definitions as-is (fast, free)
  │
  ├─ 4. If found AND L1 != "en":
  │     → Translate definitions to L1 via DeepSeek LLM
  │     → Include word metadata as context for accurate translation
  │
  └─ 5. If NOT found in any CSV dictionary:
        → Generate full entry via DeepSeek LLM
        → match_type = "llm"
```

### Match Type

Every result includes a `match_type` field so the client can display appropriate UI:

| `match_type` | Meaning | Example |
|---|---|---|
| `exact` | Word matched directly in dictionary | "食べる" hits EDICT entry |
| `lemma` | Matched via inflected/root form | "食べます" → lemma "食べる" |
| `fuzzy` | Closest match via fuzzy search | "たべる" (kana only) → "食べる" |
| `llm` | Generated by LLM, no dictionary entry | Rare word not in any CSV |

### LLM Translation Prompting Strategy

When translating English definitions to another L1, we must provide **full word context** to the LLM so translations are accurate. For example, the English definition "to eat" could refer to consuming food, corrosion, or resource consumption — the LLM needs to know it's a dictionary definition for a verb.

**Prompt template for definition translation:**

```text
You are translating dictionary definitions for a language learning app.
Translate the following definitions from English to {L1_NAME}.

Word: {head}
Language: {l2_name}
Part of speech: {part_of_speech}
Level: {level_description}  (e.g., "JLPT N5", "HSK 1", "CEFR A2")
Context: This word appears in video subtitles. The user is a {l1_name} speaker learning {l2_name}.

English definitions:
{definition_list}

Return ONLY a JSON array of translated definitions, same order, same count.
Example: ["comer", "tomar una comida"]
```

**Prompt template for LLM-generated entry (no CSV):**

```text
You are a dictionary for a language learning app. Generate a structured entry for the following word.

Word: {text}
Language: {l2_name} (ISO 639-1: {l2_code})
User's language: {l1_name} (ISO 639-1: {l1_code})
Context: This word appears in the subtitle: "{context_line}"

Return ONLY valid JSON with these fields:
{
  "head": "the word in its canonical/dictionary form",
  "definitions": ["definition 1 in {l1_name}", "definition 2 in {l1_name}"],
  "pronunciation": "phonetic guide in Latin script or IPA",
  "part_of_speech": "noun|verb|adjective|adverb|particle|...",
  "level_guess": "A1|A2|B1|B2|C1|C2"  (your best guess at CEFR level)
}
```

### Unified Response Schema

```typescript
interface LookupResponse {
  query: {
    text: string;          // original user query
    l2: string;            // ISO 639-1
    l1: string;            // ISO 639-1
  };
  results: WordEntry[];
}

interface WordEntry {
  // ── Core (always present) ──
  id: string;                    // canonical ID for save/bookmark
  head: string;                  // primary display form in L2 script
  definitions: string[];         // definitions in the requested L1
  pronunciation: string;         // phonetic guide (Latin-script)
  match_type: "exact" | "lemma" | "fuzzy" | "llm";

  // ── Optional metadata ──
  part_of_speech?: string;
  level?: {
    scale: "hsk_2010" | "hsk_2026" | "cefr" | "jlpt";
    value: number | string;
  };
  frequency?: number;            // 0-1 normalized rank

  // ── Language-specific scripts ──
  alternate?: string;            // alternate script: traditional, kana, hanja, hantu, hangul, han
  han_script?: {
    traditional?: string;
    simplified?: string;
    kanji?: string | null;
    hanja?: string | null;
    hangul?: string;
    han?: string;
    hantu?: string;
  };
  phonetic_detail?: {
    pinyin?: string;
    pinyin_numeric?: string;
    kana?: string;
    romaji?: string;
    jyutping?: string;
    romanization?: string;
    ipa?: string;
    pitch_accent?: number[];
    stressed?: string;
  };

  // ── Source info ──
  source: "hsk-cedict" | "cc-canto" | "edict" | "kengdic" | "klingonska" | "wiktionary" | "llm";
}
```

## CSV Data Standard

All dictionary source CSVs under `zerotohero-python/data/dictionaries/` follow a unified 8-column schema, with optional language-specific extra columns appended after column 8.

### Common Columns (every dictionary)

| # | Column | Type | Description |
|---|---|---|---|
| 1 | `id` | string | Unique ID, prefixed by source (`cedict-…`, `edict-…`, `canto-…`) |
| 2 | `head` | string | Primary display form in L2's main script |
| 3 | `alternate` | string | Alternate script form (or empty) |
| 4 | `pronunciation` | string | Latin-script phonetic guide |
| 5 | `definitions` | string | Pipe-separated (`\|`) English definitions |
| 6 | `part_of_speech` | string | Language-specific POS (or empty) |
| 7 | `level` | string | Encoded as `scale:value` (e.g., `hsk_2026:1`, `jlpt:N5`) or empty |
| 8 | `frequency` | float | Normalized 0–1 frequency rank (or empty) |

### Per-Language Column Mapping

| # | Column | zh (HSK CEDICT) | yue (CC-Canto) | ja (EDICT) |
|---|---|---|---|---|
| 1 | `id` | `hskId` | md5-derived | EDICT `id` |
| 2 | `head` | `simplified` | `traditional` | kanji (or kana if no kanji) |
| 3 | `alternate` | `traditional` | `simplified` | kana (or empty if kana-only) |
| 4 | `pronunciation` | `pinyin` | `jyutping` | romaji |
| 5 | `definitions` | `/`→`\|` split | `/`→`\|` split | already `\|`-separated |
| 6 | `part_of_speech` | guessed from def | guessed from def | raw EDICT POS codes |
| 7 | `level` | `hsk_2026:N` or `hsk_2010:N` | *(empty)* | *(empty)* |
| 8 | `frequency` | from `weight` | *(empty)* | *(empty)* |

### Language-Specific Extra Columns

Appended after the 8 common columns:

| Language | Extra Column | Description | Example |
|---|---|---|---|
| `ja` | `pitch_accent` | Comma-separated accent numbers | `0,2` |
| `yue` | `mandarin_pinyin` | Mandarin pinyin cross-reference | `yi1 jian4…` |
| `ru` | `stressed` | Accented Cyrillic form (stress marks) | `молоко'` |

### ISO Code Convention

Wiktionary CSV filenames use **ISO 639-1** (2-letter) codes wherever a mapping exists (e.g., `ru.csv`, `zh.csv`). For languages without an ISO 639-1 code, the ISO 639-3 (3-letter) code is used (e.g., `aaa.csv`). No two ISO 639-3 codes map to the same ISO 639-1 code, so there are no filename collisions.

### Design Rationale

- **`pronunciation` is always Latin-script** — pinyin, romaji, jyutping. Native-script phonetics (kana) go in `alternate` or `phonetic_detail` in the loader output.
- **`definitions` always pipe-separated** — avoids ambiguity with commas in definitions.
- **`level` is a single encoded string** — `scale:value` format supports any proficiency framework without adding columns.
- **Extra columns are language-specific** — the loaders know their own CSV shape; extra columns carry data that doesn't fit the common model (pitch accent, cross-ref pinyin).

## Lemmatizer Schema

The unified `/lemmatize` endpoint accepts text and a language code, returning a language-agnostic token array.

### Endpoint

```
POST /lemmatize  { "text": "я люблю молоко", "l2": "ru" }
```

### Response Schema

```typescript
interface LemmatizeResponse {
  tokens: LemmatizedToken[];
}

interface LemmatizedToken {
  text: string;              // surface form in the original text
  lemmas: Lemma[];           // empty array = punctuation/spaces (no lookup)
  pronunciation?: string;    // phonetic guide (rare — Arabic only)
}

interface Lemma {
  lemma: string;             // base/dictionary form for lookup
  part_of_speech?: string;   // language-specific POS tag
  pronunciation?: string;    // lemma-level pronunciation (Arabic)
}
```

### Design Decisions

- **`lemmas` is always an array** — handles homographs like French `lit` → `lire` (to read) | `lit` (bed), English `saw` → `see` | `saw` (noun)
- **`part_of_speech`** (not `pos`) — consistent with `DictionaryEntry`
- **Flat list, no nesting** — one token = one word. No tree structures needed.
- **Punctuation: `lemmas: []`** — frontend checks `lemmas.length > 0` to decide whether to look up in dictionary
- **Fallback** — languages without a lemmatizer get a simple word/punctuation split

### Language → Lemmatizer Mapping

| Languages | Lemmatizer | Notes |
|---|---|---|
| `zh` | Jieba (pseg) | POS + pinyin pronunciation |
| `ja` | MeCab | POS + kana pronunciation |
| `ko` | OpenKoreanText | Stem-based lemmas |
| `ru` | Pymorphy2 | Multi-lemma with morphology |
| `ar` | Qalsadi | Multi-lemma alternatives + pronunciation |
| `fa` | Hazm | POS-tagged |
| `tr` | Zeyrek | POS-tagged |
| `my` | Pyidaungsu | POS-tagged |
| `es,fr,de,...` | spaCy | 20+ languages |
| `cs,hu,sv,...` | Simplemma | 25+ languages |
| `*` (other) | Fallback regex | Word/punctuation split only |

## Implementation Plan

### Phase 4a: Backend Endpoint

1. Create `zerotohero-python/app_dictionary.py` with `/dictionary/lookup` POST endpoint
2. Build `dictionary_loaders/base.py` — abstract loader with CSV loading + memory caching
3. Build loader for each L2: `cedict.py`, `edict.py`, `kengdic.py`, `open_russian.py`, `wiktionary.py`
4. Implement search logic: exact → lemma → fuzzy
5. Integrate DeepSeek for definition translation (context-aware prompt)
6. Integrate DeepSeek for full word generation (no-CSV fallback)

### Phase 4b: Next.js Hook + Components

7. Create `useDictionary()` hook in `apps/web/src/hooks/`
8. Create `DictionaryPopup` component (shadcn/ui Sheet)
9. Make subtitle words clickable in `SubtitleDisplay`
10. Wire token click → `lookupWord()` → `DictionaryPopup`

### Phase 4c: Dictionary Search Page

11. Create `/dictionary` search page with search input + results list
12. Add `/dictionary/word/[id]` detail page
13. Add word saving/bookmarking via Directus `user_data`

## Consequences

### Positive
- Any L1 is supported natively — not just English
- No large downloads for web users
- LLM definitions are contextual and rich
- CSV dictionaries provide authoritative, consistent data for common words
- Architecture easily extends to new L2 languages (just add a CSV loader or fall through to LLM)

### Negative
- Requires network — no offline use (acceptable: GO app handles offline)
- ~1-3s latency for LLM-translated or LLM-generated entries
- API cost for LLM calls (mitigated by: CSV cache for common words, result caching)
- LLM definitions may be less authoritative than curated dictionaries for rare words

### Risks
- LLM hallucination for very rare words → mitigated by showing `source: "llm"` badge in UI
- DeepSeek API downtime → mitigated by returning English definitions as fallback
- CSV files changing format → mitigated by version pinning
