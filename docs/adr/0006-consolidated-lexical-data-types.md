
# ADR 0006: Consolidated Lexical Data Types

> **Date:** 2026-07-14
> **Context:** Unifying words, phrases, and LLM-generated entries into a single type system

---

## Decision

Consolidate all user-saved vocabulary into a single `LexicalItem` type, replacing four independent stores. The type discriminates on `source` (dictionary entry vs. freeform text) rather than on user action (save word vs. save phrase vs. bookmark hit).

---

## Motivation

Classic maintains four separate Vuex stores (`savedWords`, `savedPhrases`, `savedCollocations`, `savedHits`) with different shapes, dedup logic, and sync paths. This causes fragmentation, and inflexibility because users can't mix 'words' (those with dictionary entry) and 'phrases' those without. Having a unified scheme also allows the user to save multiple encounters of a word under the same lemma.

A unified `LexicalItem` type with `instances` (individual occurrences in context) is more elegant.

---

## Types

### Base Entry Shape

`LexicalEntry` is the common shape shared by dictionary entries and LLM-generated entries. It contains the fields needed for display in a saved-words list or dictionary popup.

```ts
/**
 * Common shape for any lexical lookup result — whether from a curated dictionary
 * (EDICT, CEDICT, etc.) or generated by an LLM at query time.
 */
interface LexicalEntry {
  /** Canonical/dictionary form of the word or phrase in the target language's script.
   *  Example: '食べる' (Japanese), 'run' (English), '你好' (Chinese). */
  head: string;

  /** One or more definitions in the user's L1 (or English fallback).
   *  Ordered by relevance. Single-language — for multi-language translations,
   *  see LexicalItem.translations. */
  definitions: string[];

  /** Phonetic guide in Latin script or IPA.
   *  Example: 'taberu' (romaji), 'nǐ hǎo' (pinyin). */
  pronunciation: string;

  /** Proficiency level(s) if known. Scale depends on the language:
   *  Chinese → hsk_2010 and/or hsk_2025, Japanese → jlpt, others → cefr.
   *  A word may have multiple levels across different scales (e.g., both
   *  hsk_2010:3 and hsk_2025:2). The UI displays the most relevant one
   *  based on user preference; when both are present, hsk_2025 is shown
   *  by default for Chinese.
   *  null or empty array means unclassified (not "beginner"). */
  levels?: { scale: string; value: number | string }[] | null;

  /** Part of speech. Language-specific values:
   *  English/European: 'noun', 'verb', 'adjective', etc.
   *  Japanese: EDICT POS codes ('v5k', 'adj-i', etc.)
   *  Chinese: inferred from definitions ('verb', 'noun', etc.) */
  part_of_speech?: string | null;
}
```

### Dictionary Entry

A lookup result from a curated, versioned dictionary (EDICT, CEDICT, CC-Canto, Kengdic, etc.). Immutable per dictionary version — the same `dictionaryId:entryId` always returns the same data.

```ts
/**
 * A canonical entry from a curated dictionary.
 *
 * Identity: { dictionary.id, dictionary.version, id }
 *   Example: { id: 'edict', version: '2019' }, entry: '2047'
 *
 * This is NOT the same as an LLM-generated entry. Dictionary entries are
 * stable, versioned, and curator-verified. LLM entries are ephemeral.
 */
interface DictionaryEntry extends LexicalEntry {
  kind: 'dictionary';

  /**
   * The dictionary that owns this entry.
   *
   * Why scoped IDs: Classic used composite IDs like 'edict-2047'. This breaks
   * when EDICT 2024 renumbers entries. Scoping lets us migrate:
   *   Old: { dictionary: 'edict_2019', id: '2047' }
   *   New: { dictionary: 'edict_2024', id: '3192' }
   * The LexicalItem.source is updated; instances are preserved.
   *
   * For the LLM pseudo-dictionary:
   *   { id: 'llm', name: 'AI-Generated', version: 'deepseek-v4-flash' }
   * The version is the model name — changing models invalidates the cache.
   */
  dictionary: {
    id: string;      // 'edict', 'cedict', 'cc-canto', 'kengdic', 'llm'
    name: string;    // 'Edict', 'HSK CEDICT', 'AI-Generated'
    version: string; // '2019', '2021', 'deepseek-v4-flash'
  };

  /** Entry ID scoped within this dictionary version.
   *  Example: '2047' (not the Classic composite 'edict-2047').
   *  For LLM entries, this is the query text. */
  id: string;

  /**
   * How this entry was matched to the user's query.
   *
   * 'exact'  — query text matched the headword directly (e.g., searched "食べる", got "食べる")
   * 'lemma'  — query was an inflected form; matched to the lemma (e.g., searched "食べます", lemma "食べる")
   * 'fuzzy'  — no exact or lemma match; closest fuzzy match (e.g., searched "たべる" in kana, matched "食べる")
   * null     — not applicable (dictionary browse, not a search result)
   *
   * NOTE: 'llm' is NOT a valid match_type here. LLM-generated entries use
   * the LlmGeneratedEntry type instead. */
  match_type: 'exact' | 'lemma' | 'fuzzy' | null;

  /**
   * Corpus frequency — how common this word is in the language.
   * Normalized 0–1 rank. Lower = less common.
   * Baked into the dictionary CSV tables at build time.
   */
  frequency?: number | null;

  /**
   * Zipf-derived frequency level. 1–7 integer.
   * 1 = most common words (~1 per million), 7 = rarest.
   * Used for proficiency gating and sorting. */
  frequencyLevel?: number | null;

  // ── Study material coverage ──

  /**
   * Study materials (textbooks, courses, graded readers) that cover this entry.
   * Each entry records where in the material this word appears (book, lesson, dialog).
   *
   * Separate from the dictionary itself — study materials are independent of
   * dictionary versions. The HSK Standard Course doesn't change when CC-CEDICT
   * is updated, so coverage data lives in its own SQLite table and is joined
   * at query time.
   *
   * null means no coverage data exists for this entry.
   * undefined means coverage hasn't been loaded yet (lazy loading).
   *
   * Example: A word appearing in HSK Standard Course Book 3, Lesson 18, Dialog 2:
   *   { material: 'HSK Standard Course', author: 'Jiang Liping', year: 2014,
   *     targetLevel: { scale: 'hsk_2010', level: 3 },
   *     location: { book: 3, lesson: 18, dialog: 2 } }
   */
  studyMaterials?: StudyMaterialCoverage[] | null;

  // ── Language-specific scripts ──

  /** Alternate script form. Language-dependent:
   *  Chinese: traditional (if head is simplified), or simplified (if head is traditional)
   *  Japanese: kana reading (if head is kanji)
   *  Korean: hanja (if head is hangul)
   *  Vietnamese: chữ Hán (if head is quốc ngữ) */
  alternate?: string | null;

  /** CJK script variants. Populated based on language. */
  han_script?: {
    traditional?: string;
    simplified?: string;
    kanji?: string | null;
    hanja?: string | null;
    hangul?: string;
    han?: string;     // Vietnamese chữ Hán
    hantu?: string;   // Vietnamese chữ Hán (alternate)
  } | null;

  /** Detailed phonetic information beyond the top-level pronunciation string. */
  phonetic_detail?: {
    pinyin?: string;
    pinyin_numeric?: string;
    kana?: string;
    romaji?: string;
    jyutping?: string;
    romanization?: string;
    ipa?: string;
    pitch_accent?: number[];    // Japanese pitch accent pattern
    stressed?: string;          // Stressed syllable marker
  } | null;

  // ── Classifiers ──

  /**
   * Morphological classifiers — measure words, counters, grammatical gender,
   * noun classes. Language-dependent structure discriminated by kind.
   *
   * WHY A DISCRIMINATED UNION: Different languages encode classification
   * differently. Chinese uses measure words (个/本/张) embedded in CC-CEDICT's
   * CL: tags. German uses grammatical gender (der/die/das). Japanese uses
   * counters (本/枚/匹). A single string can't represent all of these.
   *
   * Each classifier may optionally be tied to a specific sense via senseIndex
   * (different definitions of the same word may take different classifiers).
   * When senseIndex is undefined, the classifier applies to all senses.
   */
  classifier?: Classifier[] | null;

  // ── Future fields ──

  /** Audio pronunciation URLs. Multiple dialects possible.
   *  Example: [{ url: '...', dialect: 'standard' }, { url: '...', dialect: 'kansai' }] */
  audio?: { url: string; dialect?: string }[] | null;

  /** Inflected/alternate forms from dictionary morphology tables.
   *  Tags describe the morphological features.
   *  Example (Japanese): { form: '食べます', pronunciation: 'tabemasu', tags: ['polite', 'present'] }
   *  Example (Spanish):  { form: 'comí', tags: ['1s', 'preterite', 'indicative'] } */
  inflections?: {
    form: string;
    pronunciation?: string;
    tags: string[];
  }[] | null;
}
```

### Study Material Coverage

```ts
/**
 * Records that a dictionary entry is covered by a specific study material
 * (textbook, course, graded reader, etc.) at a particular location.
 *
 * Imported from CSV sources like hsk_standard_course.csv.
 * Stored in a separate SQLite table (e.g., hsk_curriculum) and joined
 * at query time, rather than baked into dictionary CSV tables.
 *
 * WHY SEPARATE TABLE: Study materials are independent of dictionary versions.
 * The HSK Standard Course doesn't change when CC-CEDICT is updated. Keeping
 * coverage data separate avoids re-importing it with every dictionary rebuild.
 *
 * Multiple StudyMaterialCoverage entries may reference the same
 * dictionary entry (e.g., a word appears in both HSK Standard Course
 * AND a separate graded reader series).
 */
interface StudyMaterialCoverage {
  /** Human-readable name of the study material.
   *  Example: 'HSK Standard Course' */
  material: string;

  /** Author or publisher.
   *  Example: 'Jiang Liping' */
  author?: string;

  /** Publication year.
   *  Example: 2014 */
  year?: number;

  /** Scale and level this material targets.
   *  Example: { scale: 'hsk_2010', level: 3 } means HSK Level 3 */
  targetLevel: {
    scale: string;
    level: number | string;
  };

  /** Location within the material — book/volume, lesson/chapter, dialog/exercise.
   *  Each field is optional because different materials organize differently. */
  location: {
    book?: string | number;
    lesson?: string | number;
    dialog?: string | number;
  };

  /** The example sentence from this material, if any. */
  example?: string;

  /** L1 translation of the example sentence. */
  exampleTranslation?: string;
}
```

### Classifier Types

```ts
/**
 * A morphological classifier — how the language categorizes this noun/word.
 *
 * Discriminated on `kind` because different language families encode
 * classification in fundamentally different ways. The common field is
 * `senseIndex` — which definition this classifier applies to (since
 * different senses of the same word may take different classifiers).
 */
type Classifier =
  | MeasureWord      // zh: 个/本/张, ja: 本/枚/匹, ko: 개/명/마리
  | GrammaticalGender // de: der/die/das, fr: le/la, es: el/la
  | NounClass;        // Swahili: ki-/vi-, Russian: animacy, etc.

/** Which definition index this classifier applies to.
 *  undefined = applies to the word in general (most common). */
type SenseIndex = number | undefined;

// ── Measure Word (Chinese, Japanese, Korean, Vietnamese, etc.) ──

/**
 * A numeral classifier / measure word.
 *
 * SOURCE: CC-CEDICT CL: tags, EDICT counter annotations, Kengdic counters.
 *
 * In CC-CEDICT, multiple CL: tags follow their respective senses inline:
 *   /chess; chess-like game/CL:盤|盘[pán]/chess piece/CL:個|个[gè]/
 *
 * Each CL: entry becomes a MeasureWord with the sense it modifies.
 *
 * SIMPLIFIED CASE: Some CC-CEDICT entries list all CLs at the end of the
 * entry rather than per-sense. These have senseIndex = undefined.
 */
interface MeasureWord {
  kind: 'measure_word';

  /** The classifier word in traditional Chinese (or the language's traditional script). */
  traditional: string;
  /** The classifier word in simplified Chinese (or the language's simplified script).
   *  Same as traditional for Japanese counters. */
  simplified: string;
  /** Phonetic reading of the classifier (pinyin, romaji, etc.). */
  reading: string;

  /** Which definition this classifier applies to, if sense-specific. */
  senseIndex?: SenseIndex;
}

// ── Grammatical Gender (German, French, Spanish, Russian, Arabic, etc.) ──

/**
 * Grammatical gender or animacy classification.
 *
 * SOURCE: Wiktionary data, dictionary metadata.
 *
 * German: der Tisch (masculine), die Lampe (feminine), das Buch (neuter)
 * French: le livre (masculine), la table (feminine)
 * Spanish: el libro (masculine), la mesa (feminine)
 * Russian: стол (masculine inanimate), человек (masculine animate)
 * Arabic: كتاب (masculine), مدرسة (feminine)
 */
interface GrammaticalGender {
  kind: 'gender';

  /** The gender/animacy value. Language-specific:
   *  German: 'masculine' | 'feminine' | 'neuter'
   *  French/Spanish/Italian: 'masculine' | 'feminine'
   *  Dutch/Swedish: 'common' | 'neuter'
   *  Russian: 'masculine' | 'feminine' | 'neuter' + animacy handled separately
   *  Polish: 'masculine_personal' | 'masculine_animate' | 'masculine_inanimate'
   *          | 'feminine' | 'neuter' */
  value: string;

  /** Which definition this gender applies to, if the word has multiple senses
   *  with different genders (rare but possible, e.g. "le livre" (book, m.)
   *  vs "la livre" (pound, f.) in French). */
  senseIndex?: SenseIndex;
}

// ── Noun Class (Bantu languages, some Caucasian languages, etc.) ──

/**
 * Noun class — a more elaborate classification system than gender,
 * found in Bantu languages (Swahili ki-/vi-, m-/wa-), some Caucasian
 * languages, and Australian Aboriginal languages.
 *
 * SOURCE: Wiktionary data.
 *
 * Example (Swahili):
 *   kitabu (book) → noun class 7/8 (ki-/vi-)
 *   mtu (person) → noun class 1/2 (m-/wa-)
 */
interface NounClass {
  kind: 'noun_class';

  /** The noun class identifier. Format depends on the language:
   *  Swahili: '7/8' (singular/plural class pair)
   *  isiZulu: 'umu-/aba-' (prefix pair)
   *  Dyirbal: 'bayi' (class I), 'balan' (class II), etc. */
  value: string;

  /** Which definition this class applies to, if sense-specific. */
  senseIndex?: SenseIndex;
}
```

### Classifier Parsing

Classifiers are extracted at dictionary import time, not at query time. Each dictionary format requires its own extraction logic:

```
CC-CEDICT (Chinese):
  Input:  /book/CL:本[ben3],冊|册[ce4]/to write/
  Parse:  Split on / → find CL: patterns → extract traditional|simplified[pinyin]
  Output: [{ kind: 'measure_word', traditional: '本', simplified: '本', reading: 'ben3' },
           { kind: 'measure_word', traditional: '冊', simplified: '册', reading: 'ce4' }]

EDICT (Japanese):
  Input:  definitions may include counter annotations like "counter for X"
          or the entry itself IS a counter word (POS tag includes 'ctr')
  Output: [{ kind: 'measure_word', traditional: '本', simplified: '本', reading: 'hon' }]
          (Japanese counters use the same MeasureWord shape — traditional = simplified)

Wiktionary (German/French/etc.):
  Input:  POS field or definitions include gender markers like "{m}", "{f}", "{n}"
  Output: [{ kind: 'gender', value: 'masculine' }]
```

The parsed classifiers are stored as a JSON array in a `classifiers TEXT` column in SQLite, keyed by the entry ID.

### Study Material Parsing

Study material coverage is imported from CSV sources at dictionary build time. Each source maps headwords to locations within a specific textbook or course.

```
HSK Standard Course (hsk_standard_course.csv):
  Input:  simplified,traditional,book,lesson,dialog,example,exampleTranslation
          不但……而且……,不但……而且……,3,18,3,"他不但听到，而且也看到了。","He not only heard..."
  Parse:  Match headword → map book/lesson/dialog → build StudyMaterialCoverage
  Output: { material: 'HSK Standard Course', author: 'Jiang Liping', year: 2014,
            targetLevel: { scale: 'hsk_2010', value: 3 },
            location: { book: 3, lesson: 18, dialog: 3 },
            example: '他不但听到，而且也看到了。',
            exampleTranslation: 'He not only heard it, but he saw it, too.' }

Future sources (JLPT prep books, Integrated Chinese, Genki, etc.):
  Each source gets its own CSV with language-appropriate columns.
  The StudyMaterialCoverage shape is source-agnostic — any textbook can
  map its location scheme to { book, lesson, dialog }.
```

Coverage data is stored in a separate SQLite table (e.g., `hsk_curriculum`) keyed by the dictionary entry ID. At query time, coverage rows are joined to the dictionary entry and returned as `studyMaterials`.

### Future fields ──

When no dictionary has the word, or the user's L1 has no dictionary, the Python backend generates a definition via DeepSeek LLM. Unlike a `DictionaryEntry`, this is **not canonical** — the same word queried in different contexts may produce different definitions.

```ts
/**
 * A non-canonical, context-dependent entry generated by an LLM.
 *
 * These are ephemeral by nature:
 *   - The same word queried with different context sentences may get different definitions
 *   - Model updates may change the output
 *   - There is no curator verification
 *
 * CURRENT LIMITATION (noted in "Future Work"):
 *   llmEntry currently lives on LexicalItem (shared across all instances).
 *   It should move to Instance.llmEntry so each occurrence can have its own
 *   context-specific definition. This is deferred to avoid schema churn
 *   while the LLM integration matures.
 */
interface LlmGeneratedEntry extends LexicalEntry {
  kind: 'llm';

  /** The model that generated this entry.
   *  Used for cache invalidation — if the model changes, entries are stale.
   *  Example: 'deepseek-v4-flash' */
  model: string;

  /**
   * The sentence provided as context in the LLM prompt.
   * This is the subtitle line or surrounding text that the word appeared in.
   *
   * Why store this: it enables re-querying with the same context if the
   * model updates, and helps users remember where they encountered the word.
   *
   * Example: "I went for a run this morning" → LLM knows "run" is a noun (exercise),
   *          not a verb (execute). */
  contextSentence?: string;

  /**
   * Corpus frequency of the headword, looked up from our frequency tables
   * at generation time. NOT generated by the LLM — it's a side lookup.
   */
  frequency?: number | null;
  frequencyLevel?: number | null;
}
```

### Lexical Item

The core user-data type. A `LexicalItem` represents a distinct vocabulary item (word, phrase, or expression) that a user has encountered and optionally saved. Its identity is determined by `source`, not by a traditional database ID.

```ts
/**
 * The source of a LexicalItem — determines its identity and where its
 * display data (definitions, pronunciation, etc.) is resolved from.
 *
 * IMPORTANT: LexicalItems do NOT have database IDs. Identity is derived
 * from source via the lexicalItemKey() utility function:
 *
 *   dict:{dictionaryId}:{entryId}  for dictionary entries
 *   text:{normalizedText}          for phrasebook text items
 *   llm:{l2}:{hash}               for LLM-generated entries
 *
 * This is because "lemma ID" is a human concept, not a machine one.
 * When dictionaries are updated and entries are renumbered, the user
 * must manually remap (update the source from one dictionary:entryId
 * to another). Instances are preserved across remappings.
 */
type LexicalItemSource =
  /**
   * The item corresponds to a specific entry in a curated dictionary.
   *
   * Example: { kind: 'dictionary', dictionaryId: 'edict', entryId: '2047' }
   *
   * Display data comes from LexicalItem.canonicalEntry (cached DictionaryEntry).
   * This is the most common case for words that exist in our dictionaries.
   */
  | { kind: 'dictionary'; dictionaryId: string; entryId: string }

  /**
   * The item is identified by its text string. Covers two sub-cases:
   *
   *   llm: true  → Word not found in any dictionary; definitions were
   *                generated by an LLM at lookup time. Display data
   *                comes from LexicalItem.llmEntry.
   *                Example: a rare dialect word, a neologism, gibberish
   *
   *   llm: false → A phrase or expression from a phrasebook that has no
   *                dictionary entry. Display data comes from
   *                LexicalItem.translations (multi-language map).
   *                Example: "おはようございます" (good morning)
   *
   * Identity key: the text string itself. This means "おはようございます"
   * saved from a phrasebook and "おはようございます" typed into the reader
   * are the SAME LexicalItem — instances are merged, not duplicated.
   */
  | { kind: 'text'; text: string; llm: boolean };

interface LexicalItem {
  /** How this item is identified. Determines identity key, display data source,
   *  and migration path. */
  source: LexicalItemSource;

  /**
   * ISO 639-1 code of the language this item belongs to.
   *
   * Required for cloud sync grouping (legacy compatibility: Classic groups
   * by L2). For dictionary items, derived from the dictionary at save time.
   * For text items, set explicitly based on the user's current L2.
   *
   * Example: 'ja' for Japanese, 'zh' for Chinese, 'fr' for French.
   */
  l2: string;

  /**
   * Cached dictionary entry. Only populated when source.kind === 'dictionary'.
   *
   * Populated at save time from the dictionary lookup API. Refreshed on render
   * if the dictionary version has changed (migration).
   *
   * Why cache: avoids N API calls when rendering the saved-words list.
   * A user with 500 saved words would need 500 dictionary lookups per page load
   * without this cache.
   */
  canonicalEntry?: DictionaryEntry;

  /**
   * Cached LLM-generated entry. Only populated when source.kind === 'text'
   * && source.llm === true.
   *
   * CURRENTLY: stored at LexicalItem level (shared across all instances).
   * FUTURE: should move to Instance.llmEntry for context-dependent definitions.
   * See LlmGeneratedEntry docs for rationale.
   */
  llmEntry?: LlmGeneratedEntry;

  /**
   * Multi-language translations. Only populated when source.kind === 'text'
   * && source.llm === false (phrasebook items).
   *
   * Keyed by BCP 47 language tag (typically ISO 639-1 code).
   * Each value is an ordered list of translations/definitions in that language.
   *
   * Example:
   *   {
   *     en: ["Good morning"],
   *     zh: ["早上好"],
   *     es: ["Buenos días", "Buen día"]  // multiple regional variants
   *   }
   *
   * At render time, the user sees the translation for their L1.
   * Falls back to 'en' if their L1 is not available.
   *
   * Separate from canonicalEntry.definitions / llmEntry.definitions because
   * those are single-language (the user's L1 at query time). translations is
   * multi-language, sourced from phrasebook CSV data.
   */
  translations?: Record<string, string[]>;

  /**
   * Individual occurrences of this lexical item in context.
   *
   * Each instance represents one place the user encountered this item:
   * a video subtitle line, a sentence in the reader, a phrasebook entry, etc.
   *
   * Multiple instances of the same item are grouped under one LexicalItem.
   * Example: "食べる" seen in 3 different videos → 3 instances.
   *
   * undefined (not empty array) means the item exists but hasn't been
   * encountered in any context yet (e.g., dictionary browse, phrasebook preview).
   */
  instances?: Instance[];
}

/**
 * A single occurrence of a LexicalItem in text.
 *
 * Captures both the surface form (how the word appeared — possibly inflected)
 * and the context (the surrounding sentence and where it came from).
 */
interface Instance {
  /**
   * The surface form — how the word actually appeared in the text.
   *
   * This is typically an inflected form. The LexicalItem's canonical/head form
   * is in the entry (canonicalEntry.head or llmEntry.head). The surface form
   * is for display purposes ("you encountered this as 'went', lemma 'go'").
   *
   * Example:
   *   LexicalItem → head: '食べる' (lemma)
   *   Instance.form → text: '食べます' (polite form as it appeared)
   */
  form: {
    text: string;
    /** Phonetic guide for this specific surface form, if different from lemma. */
    pronunciation?: string;
  };

  /**
   * The surrounding sentence and where it came from.
   *
   * undefined means this instance was saved without context (e.g., quick-save
   * from dictionary browse without a sentence).
   */
  context?: InstanceContext;
}

/**
 * Describes where and how the user encountered a lexical item.
 *
 * The sentence is always present (it's the text the user was reading/watching).
 * The origin describes what kind of content the sentence came from.
 * Origin is optional — a user could save a word from ephemeral text in the
 * reader without linking it to a persistent note/document.
 */
interface InstanceContext {
  /**
   * The sentence or subtitle line containing the lexical item.
   *
   * Always in the target language (L2). translation is the L1 equivalent,
   * typically machine-translated or from dual subtitles.
   */
  sentence: {
    /** The original sentence in L2. */
    original: string;
    /** L1 translation, if available. */
    translation?: string;
  };

  /**
   * What kind of content this sentence came from.
   *
   * undefined → ephemeral source (e.g., pasted text in reader, not saved as a note)
   * 'phrasebook' → from a curated phrasebook
   * 'note'       → from a user's saved note (reader document)
   * 'video'      → from a YouTube video subtitle line
   */
  origin?:
    | { kind: 'phrasebook'; phrasebookId: number }
    | { kind: 'note'; noteId: string; noteTitle: string }
    | { kind: 'video'; youtubeId: string; title: string; startTime: number };
}
```

### Saved Items

```ts
/**
 * A user's saved/bookmarked lexical item.
 *
 * Wraps a LexicalItem with a save timestamp. The timestamp is separate from
 * the item because:
 *   - The item may exist before the user saves it (phrasebook browse)
 *   - Multiple saves of the same item are deduped; the timestamp records
 *     when the user first bookmarked it
 *   - Instances have their own temporal ordering (by when encountered)
 *
 * Cloud sync: Grouped by LexicalItem.l2 for legacy compatibility with
 * Classic's SavedWords shape (Record<l2Code, SavedWord[]>). This also
 * enables efficient "show me my Japanese vocabulary" queries without
 * scanning all saved items.
 */
interface SavedItem {
  /** Unix milliseconds timestamp of when the user saved/bookmarked this item. */
  savedAt: number;

  /** The lexical item being saved. */
  item: LexicalItem;
}
```

### Phrasebook

```ts
/**
 * A curated collection of lexical items — what Classic calls a "phrasebook."
 *
 * Phrasebooks are created by admins/curators and contain ordered lists of
 * phrases or vocabulary items. Each phrasebook belongs to a language (L2).
 *
 * The special id 'saved' is used for the synthetic phrasebook constructed
 * from a user's SavedItems — this lets the saved-phrases page reuse the
 * same PhrasebookComp UI component.
 *
 * Phrases are stored as CSV in Directus (legacy). On load, each CSV row
 * is converted to a LexicalItem with:
 *   source: { kind: 'text', text: row.phrase, llm: false }
 *   translations: { en: [row.en], zh: [row.zh], ... }  // from CSV columns
 *   l2: phrasebook.l2
 */
interface Phrasebook {
  /** Directus item ID, or 'saved' for the synthetic user-saved-phrases view. */
  id: number | 'saved';

  /** Display title. May contain {l2} placeholder for language name interpolation.
   *  Example: "Essential {l2} Phrases" → "Essential Japanese Phrases" */
  title: string;

  /** HTML description, often with attribution/source links. */
  description?: string;

  /** The lexical items in this phrasebook, in display order. */
  items: LexicalItem[];

  /** Collection-level metadata. */
  meta: {
    /**
     * If set, limit subtitle search results to this TV show.
     * Used when a phrasebook is associated with a specific show
     * (e.g., "Friends S01 Phrases" → only search Friends subtitles).
     */
    tvShow?: string;

    /**
     * If true, subtitle search matches entire punctuation-delimited segments
     * rather than substrings. Classic calls this `exact`.
     *
     * Example: searching for "thank you" with exactMatch=true matches
     * the line "Thank you. Goodbye." (segment "Thank you" matches ^thank you$)
     * but NOT "thank you very much" (segment "thank you very much" ≠ "thank you").
     *
     * Used for phrases that should only match as complete expressions,
     * not as substrings of longer phrases.
     */
    exactMatch?: boolean;
  };
}
```

### Identity Utility

```ts
/**
 * Computes the canonical identity key for a LexicalItemSource.
 *
 * Used for dedup, indexing, and lookup in saved-items stores.
 *
 * Dictionary items:  'dict:{dictionaryId}:{entryId}'
 *   Example: 'dict:edict:2047'
 *   Stable across dictionary version changes (version is not in the key).
 *   Migration from edict_2019:2047 → edict_2024:3192 changes the key.
 *
 * Text items (phrasebook):  'text:{normalizedText}'
 *   Example: 'text:おはようございます'
 * LLM items:  'llm:{l2}:{hash}'
 *   Example: 'llm:zh:a1b2c3d4e5f6'
 *   Normalization is critical: fullwidth/halfwidth variants, case folding,
 *   and Unicode normalization must be applied or duplicates will occur.
 *
 * Both forms use ':' as the delimiter because it cannot appear in
 * dictionary IDs, entry IDs, or NFC-normalized text.
 */
function lexicalItemKey(source: LexicalItemSource, l2?: string): string {
  switch (source.kind) {
    case 'dictionary':
      return `dict:${source.dictionaryId}:${source.entryId}`;
    case 'text':
      if (source.llm) {
        // LLM-generated entries use a content hash keyed by language
        const hash = md5(`${l2}:${normalizeText(source.text)}`);
        return `llm:${l2}:${hash}`;
      }
      return `text:${normalizeText(source.text)}`;
  }
}

/**
 * Normalizes text for identity comparison.
 *
 * Applies: NFC normalization, NFKC-compatible fullwidth→halfwidth conversion
 * for CJK punctuation, trim, and (for non-CJK scripts) lowercasing.
 *
 * This ensures that おはよう (fullwidth) and ｵﾊﾖｳ (halfwidth) produce the
 * same key, preventing duplicate LexicalItems.
 */
function normalizeText(text: string): string {
  return text.normalize('NFC').trim();
  // Full implementation may include additional CJK-aware normalization
}
```

## Render Logic: How Display Data Is Resolved

The three source kinds resolve display data from different fields:

```ts
function resolveDisplayData(item: LexicalItem): LexicalEntry {
  switch (item.source.kind) {
    case 'dictionary':
      // Stable, canonical. Use the cached entry.
      if (!item.canonicalEntry) throw new Error('Dictionary item missing canonicalEntry');
      return item.canonicalEntry;

    case 'text':
      if (item.source.llm) {
        // LLM-generated. Currently from item-level cache; future: per-instance.
        if (!item.llmEntry) throw new Error('LLM item missing llmEntry');
        return item.llmEntry;
      }
      // Phrasebook item. Build a synthetic LexicalEntry from translations.
      return {
        kind: 'llm', // or a new 'phrasebook' kind in the future
        head: item.source.text,
        definitions: getTranslationForUserL1(item.translations),
        pronunciation: '',
        levels: null,
        part_of_speech: null,
        model: '',
        frequency: null,
        frequencyLevel: null,
      };
  }
}

/** Select the right translation for the current user's L1, falling back to English. */
function getTranslationForUserL1(
  translations: Record<string, string[]> | undefined,
  userL1: string
): string[] {
  if (!translations) return [];
  return translations[userL1] ?? translations['en'] ?? [];
}
```

## Data Flow: Save Paths

```
PATH 1: Dictionary hit
  User taps "食べる" in subtitle → lookup returns EDICT entry
  → LexicalItem {
      source: { kind: 'dictionary', dictionaryId: 'edict', entryId: '2047' },
      l2: 'ja',
      canonicalEntry: DictionaryEntry { head: '食べる', definitions: ['to eat'], ... },
      instances: [{ form: { text: '食べる' }, context: { sentence: ..., origin: { kind: 'video', ... } } }]
    }

PATH 2: LLM fallback
  User taps "zyznotawordabc" → no dictionary match → LLM generates entry
  → LexicalItem {
      source: { kind: 'text', text: 'zyznotawordabc', llm: true },
      l2: 'en',
      llmEntry: LlmGeneratedEntry { head: 'Zyznotawordabc', definitions: ['a rare goat'], ... },
      instances: [{ form: { text: 'zyznotawordabc' }, context: ... }]
    }

PATH 3: Phrasebook save
  User saves "おはようございます" from phrasebook #5
  → LexicalItem {
      source: { kind: 'text', text: 'おはようございます', llm: false },
      l2: 'ja',
      translations: { en: ['Good morning'], zh: ['早上好'], es: ['Buenos días'] },
      instances: [{ form: { text: 'おはようございます' },
                    context: { sentence: ..., origin: { kind: 'phrasebook', phrasebookId: 5 } } }]
    }
```

## What Is NOT Consolidated (Intentional)

| Concept | Reason |
|---|---|
| **Collocations** | Grammatical patterns (`"take care of"` + example), not vocabulary items. Different identity (pattern string vs. lemma), different UI. Separate type `{ pattern, example, headwordKey? }`. |
| **SRS progress** | Already has `SrsProgressStore` keyed by `lexicalItemKey()`. Stays separate — SRS is about review scheduling, not about what the item is. |
| **Subtitle search results** | Transient. Only become Instances when the user explicitly bookmarks/saves one. |
| **Notes (reader documents)** | Referenced by `InstanceContext.origin` but not owned by the lexical system. Notes have their own CRUD. |

## Migration from Classic SavedWord

```
SavedWord.id              → LexicalItem.source (parse composite → scoped)
SavedWord.forms           → Kept. Lemmatization is not always reliable for every
                            instance — saving inflected forms ensures accurate
                            highlighting in the original context. The forms list
                            maps to Instance.form.text per occurrence.
SavedWord.date            → SavedItem.savedAt
SavedWord.context.form    → Instance.form.text
SavedWord.context.text    → Instance.context.sentence.original
SavedWord.context.youtube_id  → Instance.context.origin.youtubeId
SavedWord.context.starttime   → Instance.context.origin.startTime
SavedWord.context.videoTitle  → Instance.context.origin.title

Legacy composite IDs like 'edict-2047' are split:
  prefix 'edict'  → dictionaryId
  suffix '2047'   → entryId
LLM composite IDs like 'llm-zh-abc123' are discarded:
  → source: { kind: 'text', text: originalQuery, llm: true }
  (The llmEntry is regenerated on first render if missing from migration)
```

## Future Work

1. **Per-instance LLM entries** — Move `llmEntry` from `LexicalItem` to `Instance` so each context gets its own definition. Currently deferred: the single-entry approach works for MVP, and the schema change is additive (add `Instance.llmEntry`, deprecate `LexicalItem.llmEntry`).

2. **Context-aware LLM re-query** — When staleness is detected (model version changed), re-query the LLM with the original context sentence to regenerate the entry.

3. **Level fields on phrasebook items** — Currently `levels` is only on `DictionaryEntry` and `LlmGeneratedEntry`. Phrasebook items with hardcoded levels (e.g., "JLPT N5 phrases") need `LexicalItem.levels` or should populate it in a synthetic entry.

4. **Frequency for surface forms** — Our frequency tables are lemma-only. Adding per-form frequency (e.g., how common "went" is vs. "go") would improve sorting and level estimation.

5. **Collocations** — Define the `Collocation` type and decide whether it references `LexicalItem` or stands alone.

## Consequences

- **Positive**: One save flow for all vocabulary types — words, phrases, bookmarks. Single sync path. Single UI component tree.
- **Positive**: Instance grouping enables "seen in 3 videos, 1 article" UI that no current version has.
- **Positive**: Dictionary-scoped IDs enable clean migration when dictionaries are updated.
- **Negative**: Migrating from Classic's four separate stores requires a normalization pass. Legacy data shapes (CSV phrasebooks, composite IDs) must be parsed into the new types on load.
- **Negative**: The lack of a database ID on `LexicalItem` means identity is computed, not stored. Normalization bugs (e.g., NFC vs NFKC mismatch) could cause duplicate items.
