## Context

The GO app currently uses `i18n-js` for internationalization (see `contexts/LanguageContext.tsx`). The Next.js web app uses `next-intl` (ICU MessageFormat). This creates a **syntax split** in the monorepo:

| | GO App | Next.js App |
|---|---|---|
| **Library** | `i18n-js` | `next-intl` |
| **Placeholder** | `{{key}}` or `%{key}` | `{key}` |
| **Standard** | Mustache-style (proprietary) | ICU MessageFormat |
| **Pluralization** | Not supported | `{n, plural, one {...} other {...}}` |
| **CSV source** | `lp3-trans-*.csv` (separate) | `translations.csv` (shared) |

This split has real costs:

1. **Two CSV sources of truth** — The GO app maintains its own `data/translations/lp3-trans-*.csv` files. When a new locale is added or a string is changed, both CSVs must be updated. This has already drifted for some keys.
2. **No pluralization** — `i18n-js` has no ICU plural support. Strings like `"{count} words saved"` must be handled with imperative code rather than a single ICU key.
3. **Non-standard syntax** — `{{ }}` is specific to `i18n-js`. Every other i18n library in the React ecosystem (`react-intl`, `next-intl`, `i18next`, `formatjs`) uses `{ }`.
4. **No shared translation pipeline** — The `scripts/sync-translations.mjs` CSV → JSON pipeline targets `next-intl`'s format. The GO app can't use it.

---

## Decision

**Replace `i18n-js` with `react-intl` (FormatJS) in the GO app.** This aligns both apps on ICU MessageFormat `{key}` syntax and allows the GO app to consume translations from the shared `translations.csv`.

### Key Design Decisions

| Decision | Rationale |
|---|---|
| **Keep nested JSON** | `next-intl` requires nested. `react-intl` expects flat, but a 5-line `resolveNested()` helper in `useT()` bridges this. One JSON format for both apps. |
| **Shared directory** | Both apps read from `packages/shared/locales/*.json`, generated by `sync-translations.mjs` from `translations.csv`. No per-app copies. |
| **`{{ }}` → `{ }` already done** | CSV already uses `{key}` ICU syntax (SPEC-005 did this). Zero `{{ }}` remain. Just verify with `validate-icu.mjs`. |
| **`lang.*` from web app's set** | GO's 207 language entries have visible quality issues ("AchuarName", "Kanton Achuar"). Use the web app's ~100–150 clean entries. The CSV already has these. |
| **Fix broken `lang.*` names** | ~50 obscure language names have garbled translations (LLM prompt leaks, wrong context). Fix by falling back to English via heuristics — for these languages, English IS the correct name in most locales. |
| **`find-dead-keys.mjs` as guard** | Never delete keys manually from CSV. Run the script — it scans both `apps/web/` and `language-player-3/`. A key is only dead if unused in both. |

### Why `react-intl` over alternatives

| Library | Pros | Cons |
|---|---|---|
| **react-intl** (FormatJS) | React Native support built in (v6+); full ICU; rich formatting (dates, numbers, plurals, select); mature ecosystem | Heavier bundle (~45 KB gzipped) |
| `i18next` | Popular; React Native support via `react-i18next` | ICU support requires plugin; different key format |
| `next-intl` | What web app uses | Next.js only; no React Native support |
| Keep `i18n-js` | No migration effort | No ICU; separate CSV; drift risk |

`react-intl` is the right choice because:
- It's maintained by the same team as `formatjs` (the ICU MessageFormat polyfill), ensuring correctness
- React Native support is built in (v6+)
- It supports all ICU features out of the box — no plugins, no config
- The bundle size is acceptable for a React Native app (~45 KB added vs. `i18n-js`'s ~5 KB, but eliminates the need for custom plural logic)

### What Changes

| Before (`i18n-js`) | After (`react-intl`) |
|---|---|
| `i18n.t('action.cancel')` | `intl.formatMessage({ id: 'action.cancel' })` |
| `i18n.t('msg.saved_count', { count: 5 })` with `{{count}}` | `intl.formatMessage({ id: 'msg.saved_count' }, { count: 5 })` with `{count}` |
| `i18n.locale = l1Lang.code` | `<IntlProvider locale={l1Lang.code}>` |
| `const { t } = useLanguage()` | `const intl = useIntl()` |
| Separate `lp3-trans-*.csv` → `assets/localizations/*.json` | Shared `translations.csv` → `assets/localizations/*.json` (built by `sync-translations.mjs`) |
| `{{name}}` placeholders | `{name}` placeholders |
| No plural, no select | Full ICU: `{n, plural, one {...} other {...}}`, `{gender, select, ...}` |

### ICU Features Unlocked

These are the capabilities the GO app gains that `i18n-js` can't do:

**Pluralization** (one key, all languages):
```json
"msg.saved_count": "{count, plural, =0 {No words saved} one {# word saved} other {# words saved}}"
```

**Gender/selection**:
```json
"msg.welcome": "{gender, select, male {Welcome back, sir} female {Welcome back, ma'am} other {Welcome back}}"
```

**Rich formatting** (numbers, dates):
```json
"msg.progress": "You've watched {hours, number} hours — {pct, number, percent} of your goal"
```

**Rich text messages** (React elements in translations):
```tsx
<FormattedMessage
  id="msg.contact_support"
  values={{
    link: (chunks) => <Link to="/support">{chunks}</Link>
  }}
/>
```

## Architecture

### JSON Format: Nested, Not Flat

`next-intl` (web) requires nested `{ "action": { "cancel": "Cancel" } }` — it resolves `t('action.cancel')` by traversing the object tree. `react-intl` (GO) expects flat `{ "action.cancel": "Cancel" }`. To keep one format for both apps, we keep nested JSON and bridge `react-intl` with a lightweight `resolveNested()` helper:

```ts
// hooks/use-t.ts — resolveNested bridges react-intl to nested JSON
import { useIntl } from 'react-intl';
import { useLanguage } from '@/contexts/LanguageContext';

function resolveNested(messages: Record<string, unknown>, id: string): string | undefined {
  let current: any = messages;
  for (const part of id.split('.')) {
    current = current?.[part];
  }
  return typeof current === 'string' ? current : undefined;
}

export function useT() {
  const intl = useIntl();
  const { i18n } = useLanguage();

  return (id: string, values?: Record<string, string | number>) => {
    const locale = i18n.locale;
    const message = resolveNested(i18n.translations[locale], id);
    if (!message) return id; // fallback to key name
    // react-intl handles ICU formatting ({count, plural, ...}) on the resolved string
    return intl.formatMessage({ id, defaultMessage: message }, values);
  };
}
```

This eliminates the need for `flatten-messages.ts` entirely. Both apps read the same nested JSON files.

### Provider Setup

The `IntlProviderWrapper` (Phase 1, already implemented) provides `react-intl`'s `IntlProvider` context. It sits inside `LanguageProvider` so it can read `l1Lang`:

```tsx
// contexts/IntlProvider.tsx
import React, { ReactNode, useMemo } from 'react';
import { IntlProvider } from 'react-intl';
import { useLanguage } from '@/contexts/LanguageContext';

export const IntlProviderWrapper: React.FC<{ children: ReactNode }> = ({ children }) => {
  const { l1Lang } = useLanguage();
  const locale = l1Lang?.code ?? 'en';

  // During Phases 1–2 (coexistence): pass empty messages.
  // useT() resolves from i18n.translations directly.
  // After Phase 3: IntlProvider receives the actual messages object
  // and useT() resolves via intl.formatMessage({ id }) directly.
  const messages = useMemo(() => ({} as Record<string, string>), []);

  return (
    <IntlProvider locale={locale} messages={messages} defaultLocale="en">
      {children}
    </IntlProvider>
  );
};
```

> **Note:** During Phase 3, `useT()` switches from resolving via `i18n.translations` to resolving from messages loaded directly from the shared JSON files. The `IntlProvider` will receive the actual messages object at that point, and `useT()` can drop the `resolveNested` call — `react-intl` will resolve `action.cancel` via the messages object directly because... well, it doesn't actually resolve dot-paths. So `resolveNested` stays as a thin bridge. That's fine — it's 5 lines.

### Shared Locale Directory

After Phase 2, `sync-translations.mjs` outputs nested JSON to `packages/shared/locales/`. Both apps read from here:

```
packages/shared/locales/
├── en.json          ← next-intl (web) and react-intl (GO)
├── zh-Hans.json
├── ja.json
└── ... (31 total)
```

- **Next.js**: `apps/web/src/i18n.ts` imports from `@langplayer/shared/locales`
- **GO (Metro)**: `metro.config.js` already watches `packages/` → `require('@langplayer/shared/locales/en.json')` works

### Usage in Components (Target State)

```tsx
// Before (i18n-js)
const { t } = useLanguage();
<ThemedText>{t('action.cancel')}</ThemedText>
<ThemedText>{t('msg.auto_renews_in', { days: 3 })}</ThemedText>

// After (react-intl via useT)
import { useT } from '@/hooks/use-t';
const t = useT();
<ThemedText>{t('action.cancel')}</ThemedText>                            // same
<ThemedText>{t('msg.auto_renews_in', { days: 3 })}</ThemedText>          // same
```

The call signature is **identical**. Only the import changes:
- `const { t } = useLanguage()` → `const t = useT()`
- That's it. A mechanical find-and-replace across ~100–200 files.

For components that need declarative syntax or rich text:
```tsx
import { FormattedMessage } from 'react-intl';
<FormattedMessage id="action.cancel" />
<FormattedMessage id="msg.welcome" values={{ gender: 'female' }} />
```

### Hook Wrapper

The `useT()` hook bridges `react-intl`'s flat key expectation to our nested JSON. It resolves dot-path keys via `resolveNested()`, then delegates to `intl.formatMessage()` for ICU formatting. See the full implementation in the [JSON Format section](#json-format-nested-not-flat) above.

### Phase 2: Migrate Translation Source

**Step 2a: Verify `{ }` placeholder syntax (no action needed)**

The CSV already uses `{key}` ICU MessageFormat syntax — zero `{{ }}` instances. This was done when SPEC-005 seeded the CSV from GO's data. Just verify:

```bash
# Confirm zero mustache placeholders remain
```

The `validate-icu.mjs` script (below) serves as the definitive check — if all strings parse as valid ICU, both apps are safe.

**`validate-icu.mjs`** — new script that parses every translation string with `@formatjs/icu-messageformat-parser` (the same parser both `react-intl` and `next-intl` use). Run it once after any CSV change to catch malformed ICU before JSON regeneration.

The script already produces nested `{ "action": { "cancel": "Cancel" } }` format for `next-intl` — no changes needed. The same output works for GO via the `resolveNested()` helper in `useT()`.

### Phase 2.5: Fix Mistranslated Language Names

~50 of the 207 `lang.*` entries have garbled translations for obscure languages. The common UI strings (`action.*`, `title.*`, `msg.*`, etc.) are fine — only language names are affected.

**Known patterns of bad translations:**

| Pattern | Example | Locale | Issue |
|---|---|---|---|
| Variable name leak | `AchuarName` | `ga` (Irish) | LLM prompt variable leaked into output |
| Wrong semantic context | `Kanton Achuar` | `nl` (Dutch) | "Canton" — Achuar is a language, not a Swiss canton |
| Wrong geographic name | `Malaezia` | `ro` (Romanian) | Malaysia — completely wrong for Awadhi |
| Sentence fragment | `Arama powiedział:` | `pl` (Polish) | "said:" appended — prompt fragment leaked |
| Duplicated word | `Achuar Achuar` | `sr` (Serbian) | Name duplicated |
| Wrong entity type | `Achuar Belediyesi` | `tr` (Turkish) | "Municipality" — language treated as a town |

**Root cause:** These were generated by a translation LLM without sufficient context. The model didn't know "Achuar" was a language name — it guessed based on substring patterns.

**Fix strategy — fall back to English for broken entries:**

```bash
# 1. Identify broken lang.* entries via heuristics
node scripts/find-broken-lang-names.mjs translations.csv

# 2. For each broken cell, replace with the English (en column) value
node scripts/fix-lang-names.mjs translations.csv --fallback-to-en

# 3. Validate — re-run finder, expect zero matches
node scripts/find-broken-lang-names.mjs translations.csv
```

**Heuristics for detection** (`find-broken-lang-names.mjs`):

1. **Variable leak** — contains `Name`, `Id`, `Key` suffix not present in English
2. **Sentence fragment** — contains punctuation followed by trailing words (`powiedział:`)
3. **Suspicious suffix** — ends with words like `Belediyesi`, `powiedział`, `dili` that indicate mistranslation
4. **Repeated words** — same word appears twice consecutively
5. **Geographic mismatch** — name matches a known country/city unrelated to the language

For the truly obscure languages, the English name IS the correct name in most locales. Achuar is called "Achuar" in Irish just like it is in English. Falling back to English is both safe and correct.

Only languages that have distinct names across locales (e.g., "German" → "Deutsch" → "Allemand" → "Tedesco") need real translations — and those are already correct in the CSV.

### Phase 3: Mechanical Migration

1. Replace all `const { t } = useLanguage()` with `const t = useT()` — same call signature, mechanical find-and-replace
2. Remove `i18n` and `t` from `LanguageContext.tsx` — the context no longer needs to expose them
3. Convert any remaining imperative plural logic to ICU keys (e.g., `t('msg.words', { count }) + (count === 1 ? ' word' : ' words')` → `t('msg.word_count', { count })` with ICU key)
4. Update `IntlProvider` to receive actual messages (instead of empty object) — `useT()` reads messages from the provider instead of `i18n.translations`
5. Verify: `npx tsc --noEmit` in GO app, `npx turbo typecheck` in monorepo

### Phase 4: Clean Up

1. Remove `i18n-js` dependency:
   ```bash
   cd language-player-3 && npm uninstall i18n-js
   ```
2. Remove `I18n` import and instance from `LanguageContext.tsx` — the context no longer manages translations
3. Remove `src/i18n/load-messages.ts` — superseded by direct requires from shared directory
4. TypeScript: generate type definitions for translation keys from the CSV for autocomplete:
   ```bash
   node scripts/generate-translation-types.mjs
   # → packages/shared/src/translation-keys.ts
   ```

## Consequences

### Positive
- **Single source of truth** — Both apps consume translations from `translations.csv`
- **ICU MessageFormat** — Pluralization, gender selection, number/date formatting out of the box
- **Ecosystem alignment** — `react-intl` is the standard for React i18n; community tooling, IDE plugins (i18n Ally), and linters work
- **Eliminates translation drift** — No more separate CSVs getting out of sync
- **Type-safe translations** — Can generate TypeScript types from the CSV for autocomplete on keys
- **Rich text messages** — `<FormattedMessage>` supports React elements in translations (links, bold, etc.)

### Negative
- **Bundle size increase** — `react-intl` adds ~45 KB gzipped (vs. `i18n-js`'s ~5 KB). Acceptable trade-off for ICU support and shared pipeline.
- **Migration effort** — ~100–200 files touch `useLanguage().t`. Mitigated by identical `useT()` call signature — mechanical find-and-replace only changes the import.
- **CSV placeholder migration** — `{{ }}` → `{ }` must be validated with `validate-icu.mjs` before regenerating JSONs. One-time task; the validator catches any malformed ICU strings.
- **ICU syntax learning curve** — Developers must learn ICU MessageFormat syntax (`{count, plural, ...}`). This is already required for the web app, so it's a one-time investment across the team.
